## ① 카드 對決 게임 (신규 모드)

기존 5지선다 멀티 배틀과 **별개의 모드**. 배틀 로비 상단 토글로 선택.

### 1.1 진입 흐름

```
[홈] → [멀티 배틀] → 모드 토글 (5지선다 | 카드 對決)
       → 카드 對決 선택 → "入場" 클릭 → 25% 자동 베팅 차감
       → /lobby_card 큐 등록 → 매칭 → /card_battles 방 생성
       → 證 선택 (10초) → 본 게임 → 결과 정산
```

### 1.2 베팅

- 입장 시 `S.qi * 0.25` (최소 1)를 에스크로
- 베팅 단계는 기존 4단계와 무관 — 카드 게임은 단일 25% 고정
- 매칭 실패 (75s) / 사용자 취소 시 환불

### 1.3 證 선택 단계 (10초)

- 양측 무작위 3개 證 후보 (`SYNDROMES` 26개 중에서 6개 비중복 추출, 앞 3·뒤 3 분배)
- 클라이언트 카운트다운, 시간 만료 시 자동 1번 선택
- 양측 선택 완료 → `status: 'choosing' → 'playing'` 전이 (선공이 책임)

### 1.4 카드패 구성

```js
// 카드패 = 양측 6 후보 證의 정답 처방 composition 합집합
//          + 같은 章의 다른 처방 본초 10장 (노이즈)
//          (중복 본초는 단일 카드로 정규화)
const allComp = new Set(candidates.flatMap(s => FORMULAS.find(...).composition));
const noise = [...]; // 같은 章 다른 처방의 본초, 10장 무작위
const deck = shuffle([...allComp, ...noise]);
const board = deck.splice(0, 3); // 초기 3장
```

실측 (100회 시뮬):
- **덱 크기**: min 24장, **평균 38.8장**, max 53장
- 덱 크기 변동은 작은 처방(2味 당귀보혈탕)부터 큰 처방(18味 방풍통성산)까지 후보가 다양해서 발생. 평균치는 게임 진행에 무난.

### 1.5 본 게임 (턴제)

매 턴 자기 차례에 다음 액션 가능:

| 액션 | 효과 | 비고 |
|---|---|---|
| **증상 공개** | 자기 證의 미공개 증상 1개를 상대에게 노출 | 정직 (황제 fake 적용 시 무관 증상 노출) |
| **神급 스킬** | 게임당 1회 (자기 神급 캐릭터일 때만) | 5종 (1.6 참조) |
| **전탕 시도** | 26 처방 중 1개 선택 → 보드 검증 → 정답 검증 | 정답 = 승리, 오답 = 자동 턴 종료 |
| **턴 종료** | 덱에서 1장 보드로, 상대에게 턴 넘김 | 덱 비면 추가 없음, 50턴 만료 시 무승부 |

증상 공개·스킬은 임의 횟수 가능 (정확히는 증상은 본인 증의 N개까지, 스킬은 1회까지). 전탕 시도와 턴 종료는 턴을 소비.

### 1.6 神급 캐릭터 5스킬

| 캐릭터 | 스킬 (한자/한글) | 효과 | 강도 |
|---|---|---|---|
| 黃帝 | 欺症 / 거짓 증상 | 다음 증상 공개에 자기 증과 무관한 증상 1개 공개 (자기 진짜 증상은 보존). 적이 거짓 단서로 추리 오도 | 高 |
| 神農 | 召草 / 본초 召喚 | 한자 입력 → 덱에 있으면 즉시 보드로 (없으면 효과 X) | 高 |
| 伏羲 | 卦知 / 八卦 預知 | 덱 상위 3장 자기만 미리 봄 (정보만) | 中 |
| 女媧 | 造化 / 補天造化 | 보드 본초 1장 → 덱에서 무작위 1장과 교체 | 中 |
| 岐伯 | 問難 / 雷公問難 | 자기 정답 처방 君藥 1개 보드로 (이미 있으면 효과 없이 사용 — 위약 너프) | 中 |

밸런스 의도:
- 神農·黃帝는 사용자가 명시 → 강력
- 伏羲·女媧·岐伯는 정보 또는 1장 변동 — 강력하지 않게 설계
- 모든 神급 스킬은 자기 턴에만 발동 가능, 게임당 1회
- 岐伯은 너프(이미 있으면 손해) — 자기 처방 노출 위험 vs 본초 확보 트레이드오프

### 1.7 전탕 (煎湯) 판정

```js
// 사용자가 처방 1개 선택 (모달의 "시도 가능 처방" 목록은
// 보드에 모든 composition이 있는 처방으로 사전 필터링)
function attemptDecoct(formulaId){
  const f = FORMULAS.find(x => x.id === formulaId);
  if(!f.composition.every(h => board.has(h))) return; // 본초 부족
  if(formulaId === oppSyndrome.formulaId) return WIN;
  return turnLoss;  // 오답, 자동 턴 종료
}
```

오답일 때 보드 본초는 소모되지 않음. 자기가 무엇을 시도했는지는 로그(`room.log`)에 남으므로 상대도 볼 수 있음 → 정보 노출 페널티.

### 1.8 정산

- 승자: `S.qi += bet * 2` (자기 베팅 + 상대 베팅)
- 패자: `S.qi += 0` (이미 차감)
- 무승부 (50턴 도달, `by: 'turn_limit'`): 양측 `S.qi += bet` 환불

W·L·D 기록은 `stats/records/{userId}` 에 누적 (기존 5지선다와 동일 스키마, 따라서 명예의 전당 표시에 통합됨).

## ③ 데이터 신규

### data-syndromes.js (26 증)

26 처방 1:1 매핑. 각 證은 4-5개 핵심 증상.

```js
const SYNDROMES = [
  { id: 'syn-piwi-gihu', formulaId: 'sagunja-tang',
    han: '脾胃氣虛證', ko: '비위기허증',
    symptoms: ['面色萎白','語聲低微','氣短乏力','食少便溏','脈虛弱'] },
  // ... 26개
];
```

증상은 각 처방의 `indication` 에서 핵심을 추출·축약 (게임 진행 가독성 우선). 학습용이 아닌 추리용 단서.

검증:
- FORMULAS (26) ↔ SYNDROMES (26): 1:1 완전 매핑 ✓
- 각 證 평균 5개 증상

## ⑥ 검증

- **JS syntax**: 모든 파일 `node -c` 통과
- **덱 구성 100회 시뮬**: 평균 38.8장, 최소 24장
- **SYNDROMES ↔ FORMULAS 1:1**: 누락·중복 없음
- **race condition 회피**: 방 생성·status 전이·정산 모두 단일 책임 클라이언트 (사전 순 작은 userId / 자기 _settled)

## ⑦ 알려진 제한

1. **본초 한자 입력 (神農 召草)** — 한자 IME 필요. 모바일에선 키보드 전환 번거로움. 차후 자동완성·드롭다운 보완 가능.
2. **자기 정답 증 노출 위험** — 岐伯·女媧 스킬 사용 시 추가된 본초가 자기 정답을 시사. 의도된 트레이드오프.
3. **stale 방 처리** — 5분 이상 된 방은 매칭 인식 안 함 (v6과 동일 패턴). 단 결과 화면에서 정상 종료 시 8초 후 자동 청소.
4. **본 게임 BGM** — 기존 `bgm.startBattle()` 그대로 사용. 카드 게임 전용 BGM은 v8에서 검토 가능.
5. **이순재 神階 없음** — 이순재는 神급이 아니므로 카드 對決에서 스킬 없음.
6. **모바일 화면 협소** — 보드에 본초 카드 10+ 장 동시 펼침 시 가독성 저하. CSS @media로 본초 카드 폭 54px로 축소 (480px 미만).
